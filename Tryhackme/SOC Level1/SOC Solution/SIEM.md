#### The logs source is mainly divide into two categories :
- Host-Centric Log Sources.
- Network-Centric Log Sources.
## Host-Centric Log Sources 
### These log source capture the event that occurred within or relative of the host.
### Example of host-centric log source :
- A user accessing a file
- A user attempting to authenticate.
- A process execution activity
- A process adding/editing/deleting a registry key or value.
- PowerShell execution
## Network-Centric Log Sources
### Network-related logs are generated when the hosts communicate with each other or access the internet to visit a website.
### Example of network-centric log source :
- SSH connection
- A file being accessed via FTP
- Web traffic
- A user accessing the company's resources through VPN.
- Network file sharing Activity
## Answer Nowhere
Some challenges are discussed below :
- Numerous Log Sources
- No Centralization
- Limited Context
- Limited Analysis
- Format Issues

>Logs are generated by many devices in huge numbers, stored in different places and formats, making manual investigation slow, difficult, and often missing the complete attack story.
## Why SIEM ?
Security Information and Event Management (SIEM) is a security solution that collects logs from various types of log sources, standardizes their format into a consistent one, correlates them, and detects malicious activities using detection rules.

## Alerting Process and Analysis
### Behind the Triggered Alerts
- A security solution detects threats using detection rules.
- These rules are like **"IF this happens, THEN raise an alert."**
- They help SOC analysts identify suspicious activities quickly so they can respond on time.
####  Examples of Detection Rules

- **If a user has 5 failed login attempts within 10 seconds**, raise an alert for **Multiple Failed Login Attempts**.
- **If a user successfully logs in after several failed login attempts**, raise an alert for **Successful Login After Multiple Failed Login Attempts**.
- **Whenever a USB device is connected**, raise an alert. (This is useful if the company does not allow USB devices.)
- **If outbound network traffic is more than 25 MB**, raise an alert for a **Possible Data Exfiltration Attempt**. (The limit depends on the company's security policy.)
## How is a detection rule created?
Consider the following Eventlog use cases:
## Use-Case 1: Event Log Cleared
### What happens?
Attackers often **delete Windows Event Logs** after gaining access to a system. They do this to hide their activities and make investigations harder.
Whenever someone clears the Windows Event Logs, Windows generates **Event ID 104**.
## Detection Rule

```
IF
Log Source = WinEventLog
AND
Event ID = 104

THEN 
Generate Alert:
"Event log Cleared"
```
## Use-Case 2: Detecting the `whoami` Command
### What happens?
Adversaries use  commands like `whoami` after  the exploitation/privilege escalation phase.
The following Fields will be helpful to include in the rule.
- Log source: Identify the log source capturing the event logs
- Event ID: Which ID is associated with Process Execution activity? In this case, Event ID 4688 will be helpful.
- NewProcessName: Which process name will be helpful to include in the rule?
## Detection Rule

```
IF
Log Source = WinEventLog
AND
Event ID = 4688
And
NewProcessName = whoami

THEN
Generate Alert:
"WHOAMI command Execution DETECTION"
```
## What are Field-Value Pairs?
Every log contains multiple pieces of information called fields.
### Example:

| **Field**        | **Value**         |
| :--------------- | :---------------- |
| **LogSource**    | **WinEventLog**   |
| **Event Id**     | **4688**          |
| **Username**     | **Adiminstrator** |
| **Process Name** | **whoami.exe**    |
| **Computer**     | **DESKTOP-01**    |
| **Time**         | **10:45:20**      |
## Why Normalized logs Important?
Different operation system store logs differently.
A SIEM normalized them into the common format, such as :
```
Source: Windows
Event Type: Process Execution
Process Name: whoami
```
or
```
Source: Linux
Event Type: Process Execution
Process Name: whoami
```
## Alert Investigation
 SOC analysts spend most of their time monitoring dashboard because they provide a quick summary of what's happening across the network.
 When Detection rule is triggered, an alert is generated.

The analyst then investigates the alert by:
- Looking at the related logs and events.
- Checking which detection rule was triggered.
- Finding out which condition matched.
- Deciding whether the alert is a True Positive or a False Positive.
