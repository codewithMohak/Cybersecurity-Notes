# Why Your Network Is Screaming for Help — And Nobody Is Listening

### A beginner's deep dive into SIEM: how Security Operations Centers make sense of the chaos

---

Imagine 200 devices on a corporate network. Every login, every USB plug-in, every file access, every SSH connection — all generating logs simultaneously, right now, as you read this.

Now imagine trying to find the **one log** that says _"an attacker just deleted the evidence."_

Without a SIEM, that's exactly what security teams were doing manually. And unsurprisingly, attackers were winning.

This article is about how I started learning the fundamentals of Security Information and Event Management — what a SIEM actually is, how it works inside a Security Operations Center (SOC), and the mental model shift that made everything finally click for me.

---

## The Problem: Logs Everywhere, Answers Nowhere

Every device in a network generates logs. That's not optional — it's just how modern systems work.

Your Windows workstation logs when you log in. Your Linux server logs every command executed. Your router logs every packet that leaves the network. Your VPN logs every remote connection. Your firewall logs every blocked attempt.

In a typical company network, you'll have:

- Linux and Windows workstations
- Servers
- Firewalls and routers
- VPN gateways
- Web proxies

All of them generating logs. All the time.

These logs fall into **two broad categories**:

### Host-Centric Log Sources

These capture events that happen _on_ the device itself:

- A user accessing a file
- A user attempting to authenticate
- A process executing
- A process adding, editing, or deleting a registry key
- PowerShell being executed

### Network-Centric Log Sources

These capture events that happen _between_ devices or with the internet:

- An SSH connection being made
- A file being transferred via FTP
- Web traffic
- A user connecting through VPN
- Network file sharing activity

Now here's the problem. All of these devices don't just generate _different_ logs — they generate logs in _different formats_, stored in _different locations_, with _different structures_.

A Windows Event Log looks nothing like a Linux syslog. A firewall log looks nothing like a VPN log. And nobody has the time — or honestly, the ability — to manually read through millions of log lines per day and connect the dots.

This is the core challenge:

> _Logs are generated by many devices in huge numbers, stored in different places and formats, making manual investigation slow, difficult, and often missing the complete attack story._

Five challenges sum it up perfectly:

1. **Numerous Log Sources** — too many devices generating too much data
2. **No Centralization** — logs scattered across the environment
3. **Limited Context** — one log alone rarely tells the full story
4. **Limited Analysis** — manual analysis doesn't scale
5. **Format Issues** — every device speaks a different "log language"

---

## Enter SIEM: The Central Nervous System of a SOC

Security Information and Event Management (SIEM) is a security solution that:

- **Collects** logs from all types of log sources across the environment
- **Normalizes** them into a consistent, standardized format
- **Correlates** them — connecting events across different sources
- **Detects** malicious activity using detection rules

Think of it as a translator, a librarian, and a detective all rolled into one.

### The "Log Language" Problem — Solved by Normalization

Here's something that took me a while to fully appreciate. A `whoami` command executed on Windows and a `whoami` command executed on Linux will look completely different in their raw logs. Different timestamps, different fields, different structures.

A SIEM normalizes both into something like:

```
Source: Windows
Event Type: Process Execution
Process Name: whoami
```

and

```
Source: Linux
Event Type: Process Execution
Process Name: whoami
```

Same event. Same format. Now they can be compared, correlated, and acted on.

This is called **log normalization** — and it's one of the most underrated capabilities of a SIEM. Without it, writing detection rules would be a nightmare because you'd need a different rule for every device type.

---

## How Logs Are Structured: Understanding Field-Value Pairs

Every log, regardless of where it comes from, is made up of **fields** and **values**. Think of it like a table of information attached to every event.

For example, a Windows process execution log might look like this:

|Field|Value|
|---|---|
|LogSource|WinEventLog|
|Event ID|4688|
|Username|Administrator|
|Process Name|whoami.exe|
|Computer|DESKTOP-01|
|Time|10:45:20|

These field-value pairs are what detection rules are built on. The SIEM checks these fields against conditions — and if the conditions match, an alert fires.

---

## The IF/THEN Logic of Security: Detection Rules

This is the part where everything clicked for me.

Detection rules are essentially logic statements. They say: **IF this condition is true, THEN raise an alert.**

They're how a SIEM turns raw log data into actionable security alerts. Here are some real examples:

**Multiple Failed Login Attempts:**

> If a user has 5 failed login attempts within 10 seconds → raise an alert

**Successful Login After Multiple Failures:**

> If a user successfully logs in after several failed login attempts → raise an alert _(This is a classic brute-force pattern)_

**Unauthorized USB Device:**

> Whenever a USB device is connected → raise an alert _(Useful for companies that don't allow external storage)_

**Possible Data Exfiltration:**

> If outbound network traffic exceeds 25 MB → raise an alert _(Threshold depends on the company's security policy)_

Let me walk through two real detection rule examples using Windows Event IDs.

---

### Use Case 1: Event Log Cleared — Event ID 104

Here's something clever attackers do after gaining access to a system: they **delete the Windows Event Logs**. Why? To destroy evidence of what they did — lateral movement, credential access, everything.

The good news? Windows is aware of this. Whenever someone clears the Event Logs, Windows itself generates **Event ID 104**.

So the detection rule becomes simple:

```
IF
  Log Source = WinEventLog
  AND
  Event ID = 104

THEN
  Generate Alert: "Event Log Cleared"
```

The act of hiding becomes the evidence. I find that genuinely elegant.

---

### Use Case 2: Detecting the `whoami` Command — Event ID 4688

After attackers exploit a system or escalate privileges, one of the first things they do is run `whoami` — a command that tells them _who they are_ on the system and what access they have.

Windows logs process execution under **Event ID 4688**, and includes a field called `NewProcessName` that tells you exactly what was run.

```
IF
  Log Source = WinEventLog
  AND
  Event ID = 4688
  AND
  NewProcessName = whoami

THEN
  Generate Alert: "WHOAMI Command Execution Detected"
```

This is why understanding field-value pairs matters so much. Without knowing that Event ID 4688 captures process execution and that `NewProcessName` is the field to look at, you'd never know how to write this rule.

## The SOC Analyst's Day: Alert Investigation

SOC analysts spend most of their time monitoring dashboards. These dashboards give a quick summary of what's happening across the network — active alerts, recent events, triggered rules.

When a detection rule fires, an alert is generated. The analyst's job then is to **investigate**.

That investigation involves:

1. **Looking at the related logs and events** — what actually happened?
2. **Checking which detection rule was triggered** — what condition was met?
3. **Finding out which field-value pair matched** — what specifically caused this?
4. **Deciding the verdict** — is this a **True Positive** or a **False Positive**?

A **True Positive** means the alert is real — there's actual malicious or suspicious activity. The analyst escalates, investigates further, and responds.

A **False Positive** means the alert fired but the activity is legitimate — maybe a developer ran `whoami` for debugging, or an admin intentionally cleared logs after maintenance. The analyst closes the alert and potentially tunes the rule to reduce noise.

This is where the real skill of a SOC analyst lives — not just reading logs, but _interpreting context_.

---

## The Bigger Picture

What struck me most while learning all of this is how SIEM doesn't eliminate the need for human judgment — it _enables_ it.

Without SIEM, a SOC analyst is buried under millions of unstructured log lines with no way to find signals in the noise.

With SIEM, the system handles collection, normalization, correlation, and initial detection — so the analyst can focus on what matters: investigating alerts, understanding context, and making decisions.

The detective work is still human. SIEM just makes sure the evidence is organized before the detective walks in.
